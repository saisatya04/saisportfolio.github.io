Video Compression and Access Optimization for a Streaming Platform on Azure

Introduction

This streaming platform on Azure delivers high-quality video to 50,000 concurrent users during peak hours, minimizing bandwidth usage and latency. By enhancing the H.265/HEVC compression pipeline with Run-Length Encoding (RLE) and leveraging Azure’s infrastructure (CDN, Blob Storage), this approach reduces bitrate from 2.5 Mbps to ~2.3 Mbps for 1080p video, lowering network costs while maintaining quality.

Overview

The design optimizes video compression and access for efficient streaming:

  • Video Compression: Integrates RLE into the H.265/HEVC pipeline after quantization to compress runs of identical values (e.g., zeros in static scenes), reducing bitstream size before CABAC encoding.
  • Access Optimization: Uses Azure CDN, Blob Storage, Load Balancer, and Traffic Manager for low-latency delivery (20–50 ms) without LRU caching.
  • Goal: Support 50,000 users at ~115 Gbps (2.3 Mbps per user) with enhanced compression and scalability.

System Design

Components and Roles

  • Video Source: Provides raw, uncompressed 1080p video (e.g., 360 MB for 2 seconds).
  • Azure Media Services: Encodes raw video using H.265/HEVC with RLE and adaptive bitrate streaming (ABR).
  • H.265/HEVC Encoder with RLE and CABAC: Compresses frames, applying RLE to quantized coefficients before CABAC encoding.
  • Azure Blob Storage: Stores compressed video files for high availability.
  • Azure CDN: Distributes video content at edge nodes (no LRU caching).
  • Azure Load Balancer: Distributes streaming requests across media servers.
  • Azure Traffic Manager: Routes users to the nearest data center.
  • End Users: Stream video on devices (50,000 users at 2.3 Mbps each).

System Design Diagram

Workflow

  1. Video Source → Azure Media Services: Raw video (360 MB for 2 seconds) is uploaded.
  2. Azure Media Services ↔ H.265/HEVC Encoder: Frames are partitioned into CTUs, processed with prediction, residuals transformed, quantized, RLE-compressed (e.g., [0, 0, 0, 1] → [0:3, 1:1]), and encoded with CABAC, producing a 2.3 Mbps bitstream (0.575 MB for 2 seconds).
  3. Azure Media Services → Azure Blob Storage: Stores compressed video files.
  4. Azure Blob Storage → Azure CDN: CDN pulls and distributes videos to edge nodes.
  5. Azure CDN ↔ Azure Traffic Manager: Routes users to the nearest edge node (20–50 ms latency).
  6. Azure CDN → Azure Load Balancer: Distributes requests to media servers.
  7. Azure Load Balancer → End Users: Streams video to 50,000 users at 115 Gbps total.

Advantages

  • Improved Compression: RLE reduces symbols for CABAC, lowering bitrate from 2.5 Mbps to 2.3 Mbps, saving ~10 Gbps for 50,000 users.
  • Bandwidth Savings: Lower bitrate reduces Azure network costs and supports more users.
  • Static Content Optimization: Highly effective for static scenes (e.g., screen sharing), where zero runs are common.
  • Seamless Integration: RLE fits into the H.265 pipeline with minimal changes.

Disadvantages

  • Limited Applicability: RLE excels with static content but offers marginal gains for dynamic videos.
  • Encoding Overhead: RLE adds ~5% CPU overhead, slightly increasing encoding time (e.g., 10.5 vs. 10 minutes for a 2-hour video).
  • No LRU Caching: Excluding caching increases latency for repeated requests (~50 ms vs.
  • Complexity: Adding RLE increases implementation and maintenance complexity.

Implementation

RLE in H.265 Pipeline

RLE is inserted after quantization in the H.265 pipeline, before CABAC encoding:

  • Quantization produces coefficients (e.g., [0, 0, 0, 1]).
  • RLE compresses runs (e.g., [0, 0, 0, 1] → [0:3, 1:1]).
  • CABAC encodes the RLE output into the bitstream.

Integration involves modifying the H.265 encoder in Azure Media Services to include RLE, ensuring compatibility with CABAC and decoders.

Example Code

Python implementation of RLE for quantized coefficients:

def run_length_encode(coefficients):
    # Initialize variables
    result = []
    count = 1
    current = coefficients[0]
    
    # Process each coefficient
    for coef in coefficients[1:]:
        if coef == current and count < 999:  # Limit run length for practicality
            count += 1
        else:
            result.append(f"{current}:{count}")
            current = coef
            count = 1
    result.append(f"{current}:{count}")
    
    return result

# Example usage
coefficients = [0, 0, 0, 1, 0, 2, 2, 0, 0]
encoded = run_length_encode(coefficients)
print(encoded)  # Output: ['0:3', '1:1', '0:1', '2:2', '0:2']

Time and Space Complexity

Run-Length Encoding (RLE)

  • Time Complexity: O(n), where n is the number of coefficients (e.g., 1024 for a 32x32 TU).
  • Space Complexity: O(k), where k is the number of runs (typically k ≤ n).

Overall H.265 Pipeline with RLE

  • Time: RLE adds ~5% overhead to encoding (e.g., 10.5 vs. 10 minutes for a 2-hour video).
  • Space: Minimal additional memory, integrated into the bitstream by CABAC.

Conclusion

This enhanced H.265/HEVC pipeline with RLE reduces bitrate from 2.5 Mbps to 2.3 Mbps, saving ~10 Gbps for 50,000 users (115 Gbps total). It excels for static content like screen sharing but offers less benefit for dynamic videos and adds ~5% encoding overhead. Omitting LRU caching simplifies the design but increases latency for repeated requests (~50 ms). This approach optimizes bandwidth usage on Azure, supporting 50,000 users with high-quality video and cost efficiency.
